# Decisions

- 日期：2026-06-24
- 决策：新增 `docs/SOURCE_ALIGNMENT.md`，把 Dan Koe HUMAN 3.0 原始知识库作为灵感源和审计参考，而不是可复制的产品规范。
- 原因：当前中文项目需要明确哪些内容已对齐、哪些被主动本地化或暂不纳入，避免后续 AI 误把中文 1.0 测评当作源头全量模型。
- 替代方案：只在 PRD 中简单标注灵感来源。
- 影响：后续模型、结果系统、命名和高级报告改动前，应先查阅 `docs/SOURCE_ALIGNMENT.md`；对外表达应避免暗示 Dan Koe 官方授权或完整复刻。

- 日期：2026-06-22
- 决策：新增 `docs/PRD.md` 作为 Living PRD 和单一产品事实来源。
- 原因：项目需要同时服务用户本人和后续 AI 接手，必须集中同步产品定位、新需求、已完成功能、免费版和付费版规划。
- 替代方案：继续分散维护 `PRODUCT.md`、`TODO.md`、`HANDOFF.md`。
- 影响：后续产品方向和功能状态以 `docs/PRD.md` 为准；`HANDOFF.md` 只做快速接手摘要，`TODO.md` 只做执行任务列表。

- 日期：2026-06-22
- 决策：使用根级 `app/` 目录，不使用 `src/`。
- 原因：用户指定的项目结构希望 `app/` 位于项目根目录。
- 替代方案：使用 `src/app`。
- 影响：文件路径更接近需求，Next.js 仍然完全支持。

- 日期：2026-06-22
- 决策：第一版结果保存在 localStorage，并保留 `/api/submit`。
- 原因：第一版不接数据库，但未来需要平滑迁移到 Supabase。
- 替代方案：只做纯前端，不创建 API。
- 影响：未来接入持久化时可复用服务端结果构建逻辑。

- 日期：2026-06-22
- 决策：计分规则采用可解释启发式阈值。
- 原因：需求给出方向性规则，但没有精确阈值；MVP 需要稳定可测的默认实现。
- 替代方案：让所有阈值配置化到 JSON。
- 影响：当前逻辑集中在 `lib/scoring.ts`，后续可迁移为配置驱动。

- 日期：2026-06-22
- 决策：页面不展示原始分数。
- 原因：需求要求结果页展示状态而不是分数，减少用户过度量化。
- 替代方案：显示分数和雷达图。
- 影响：UI 更克制，调试时可通过测试和内部对象查看分数。

- 日期：2026-06-22
- 决策：视觉采用浅色、克制、低饱和配色。
- 原因：产品定位是自我发展评估，不是娱乐测试或夸张营销页。
- 替代方案：暗色仪表盘或强渐变视觉。
- 影响：中文阅读体验更平稳，移动端负担更低。

- 日期：2026-06-22
- 决策：移除 `next/font/google`，改用系统中文字体栈。
- 原因：本地 Turbopack 开发服务器出现字体内部模块解析错误；系统字体对中文体验也更稳定。
- 替代方案：切换 dev 脚本到 webpack，或继续排查 Turbopack 字体依赖。
- 影响：减少开发环境不稳定性，不再依赖 Google 字体加载。

- 日期：2026-06-22
- 决策：新增 `pnpm validate:data` 数据校验命令。
- 原因：题库和模型数据未来会频繁调整，需要一个低成本检查入口保证题量、字段和覆盖范围不漂移。
- 替代方案：只依赖人工检查或测试用例间接发现。
- 影响：产品人员改 JSON 后可以先运行数据校验，再交给开发者接入。

- 日期：2026-06-22
- 决策：首页产品文案移入 `data/site-content.json`。
- 原因：减少组件中的大段文案硬编码，让产品文案和展示组件解耦。
- 替代方案：继续写在 `components/AssessmentIntro.tsx`。
- 影响：首页文案可由产品人员直接改 JSON，组件更偏展示职责。

- 日期：2026-06-22
- 决策：答题页移动端增加底部安全留白。
- 原因：本地开发环境的 Next.js 开发工具浮层可能遮挡左下角“上一题”按钮。
- 替代方案：忽略开发环境浮层，或要求手动关闭浮层。
- 影响：移动端和本地验收时底部按钮更不容易被遮挡。

- 日期：2026-06-22
- 决策：免费结果系统升级为 Metatype + Lifestyle Archetype + Core Problem + Cross-Quadrant Dynamics + Immediate Next Action。
- 原因：Living PRD 明确免费版需要更像可传播测评，结果要有记忆点、能说中当前卡点，并给出 24 小时内可执行动作。
- 替代方案：继续只展示 Human 阶段、主导象限、限制象限和 7/30/90 天建议。
- 影响：结果页和分享卡片更有传播性；结果模板 JSON 字段变多，后续需要继续审校文案。

- 日期：2026-06-22
- 决策：结果页读取 localStorage 中的答案，并用当前模板重建结果。
- 原因：结果模板会继续迭代，如果只读取旧的结果对象，浏览器缓存里的旧结构会缺少新字段。
- 替代方案：清空旧结果或继续直接展示已保存 result。
- 影响：模板升级后旧用户结果可以自动迁移到新报告结构；如果 localStorage 损坏，结果页会提示重新评估。

- 日期：2026-06-22
- 决策：第一版分享卡片下载使用浏览器 Canvas 生成 PNG。
- 原因：不需要新增依赖或服务端渲染，适合当前静态 MVP；生成尺寸固定为 1080x1440，方便社交平台分享。
- 替代方案：引入 html-to-image 类库，或未来用服务端 OG 图片生成。
- 影响：当前下载能力可离线运行；视觉复杂度受 Canvas 绘制逻辑限制，后续可替换为更专业的图像生成方案。

- 日期：2026-06-22
- 决策：第一版分享链接使用 URL 答案码 `/result/share?a=v1.<48 digits>`。
- 原因：当前不接数据库，但产品需要真实可分享结果链接来验证传播；48 个答案可稳定重建结果。
- 替代方案：继续只支持本地结果，或提前接入 Supabase。
- 影响：无需后端即可分享结果；URL 中包含答案码，不适合作为长期隐私方案，未来应由 `/result/[id]` 数据库短链接替代。

- 日期：2026-06-22
- 决策：将 `pnpm validate:data` 升级为严格数据守门，并新增 `pnpm check` 作为本地完整验收命令。
- 原因：产品人员未来会频繁调整题库、阶段和结果文案，必须在代码运行前发现字段漂移、重复项、阶段缺失、模板占位符缺失和受保护名称误用。
- 替代方案：引入第三方 JSON schema 库，或只依赖人工审查。
- 影响：当前无需新增依赖即可获得更强数据保护；后续接入 CI 时可直接运行 `pnpm check`。

- 日期：2026-06-22
- 决策：引入 Playwright 端到端测试，并让测试服务使用独立端口 `3100`。
- 原因：核心用户路径包含浏览器状态、localStorage、路由跳转和分享链接，单元测试无法证明完整体验可用。
- 替代方案：继续依赖人工浏览器验收，或只用 Vitest 测组件。
- 影响：`pnpm check` 会更接近真实交付验收；首次运行需要安装 Chromium 测试浏览器。

- 日期：2026-06-22
- 决策：新增 GitHub Actions CI，并在 CI 中运行同一条 `pnpm check`。
- 原因：项目需要长期交接和多人协作，必须让数据校验、单元测试、lint、构建和端到端测试在远程仓库自动执行。
- 替代方案：只保留本地检查命令，或把 CI 拆成多个独立 job。
- 影响：推送和 PR 到 `main` 会自动验收项目；首次远端运行需要观察 Playwright Chromium 安装和缓存情况。

- 日期：2026-06-22
- 决策：端到端测试继续覆盖异常状态和 API 路由，而不只覆盖完整答题 happy path。
- 原因：无本地结果、无效分享链接和提交 API 是第一版最容易在重构中被误伤的交接边界。
- 替代方案：只在单元测试里覆盖 API 结果生成，或保留人工验收。
- 影响：E2E 数量从 4 个增加到 7 个，CI 对真实用户路径和预留接口的保护更强。

- 日期：2026-06-22
- 决策：分享卡片下载 E2E 校验实际 PNG 文件，而不是只校验按钮点击。
- 原因：Canvas、Blob、临时链接和浏览器下载行为任一环节失败，都可能让按钮存在但用户拿不到图片。
- 替代方案：只检查“下载 PNG”按钮可见，或只单测 Canvas 生成函数。
- 影响：端到端测试会验证建议文件名、文件大小和 PNG 签名；移动端下载体验仍需真机验收。

- 日期：2026-06-22
- 决策：localStorage 读取必须做结构校验，不再只依赖 JSON 解析成功。
- 原因：缓存可能来自旧版本、手工修改或浏览器异常状态，错误结构会导致答题页或结果页崩溃。
- 替代方案：继续在组件里直接使用缓存对象，或只捕获 JSON parse 错误。
- 影响：损坏缓存会被视为无缓存，答题页回到第 1 题，结果页提示重新评估；E2E 已覆盖脏进度和脏结果。

- 日期：2026-06-22
- 决策：建立 `docs/RELEASE_1_0.md` 作为 1.0 候选版发布门槛，并将内容审校记录写入 `docs/CONTENT_REVIEW.md`。
- 原因：用户目标已从继续完善项目升级为做出 1.0 版本，需要明确 1.0 范围、发布门槛和剩余确认项。
- 替代方案：继续只用 PRD 和 TODO 追踪下一步任务。
- 影响：项目现在有明确 1.0 候选发布清单；正式升级到 `1.0.0` 前需完成真实用户反馈、真实样例阈值复核、真机检查和远程 CI 首次运行确认。

- 原因：模拟画像覆盖了极低、极高、均衡中段、单象限短板和高分失衡等典型组合，当前输出符合产品直觉；贸然微调阈值会降低可解释性。

- 影响：当前计分规则保持稳定，并通过 `tests/scoring-calibration.test.ts` 防止后续无意改变阶段分布；真实用户或产品团队样例仍作为发布前复核项。

- 日期：2026-06-22

- 原因：这两项依赖外部用户和真实设备，不能由自动测试替代；必须用统一记录格式和通过标准降低交接损耗。
- 替代方案：只在 TODO 中保留一句待办，或把反馈记录混写进开发日志。
- 影响：`docs/USER_FEEDBACK_PLAN.md` 和 `docs/MOBILE_QA_CHECKLIST.md` 成为 1.0 发布前收集外部证据的入口；完成后需把结论回填到发布清单、内容审校和阈值校准文档。

- 日期：2026-06-22
- 决策：将本机移动端 Chromium 验收作为 1.0 候选的补充证据，但不替代真实手机验收。
- 原因：自动化移动视口能稳定检查完整流程、横向溢出、刷新恢复、结果页和分享链路；但 iPhone Safari、Android Chrome 的下载和浏览器工具栏行为仍需要真实设备确认。
- 替代方案：只依赖 Playwright E2E，或把本机 Chromium 验收等同于真机验收。
- 影响：新增 `docs/MOBILE_QA_REPORT.md` 记录本机证据；`docs/RELEASE_1_0.md` 仍保留真实手机验收作为发布前确认项。

- 原因：远程拒绝信息明确指出当前 OAuth App 缺少 `workflow` scope；移除 `.github/workflows/ci.yml` 虽可绕过推送限制，但会降低 1.0 交接和远程验收质量。
- 替代方案：删除 workflow 后推送，或重写历史拆出 CI 配置。
- 影响：本地继续保留 CI 配置；远程首跑需要使用具备 `workflow` scope 的 GitHub 凭据重新推送。

- 日期：2026-06-24
- 决策：1.0 只补源头对齐边界说明，不大改题库、不改评分、不承接原文高级系统。
- 原因：当前 1.0 的目标是公开测试和交接维护，必须说明它是受 Dan Koe HUMAN 3.0 启发的中文评估工具化改编，而不是源头完整复刻或官方测评。

- 日期：2026-06-24
- 决策：UI v2 落地只整合静态原型的视觉、布局和交互模式，不改评分、题库或结果生成链路。
- 原因：原型目标是提升体验质感，产品核心仍依赖现有 JSON 数据、localStorage 流程、`result-builder` 和分享逻辑；混改业务规则会增加 1.0 发布风险。
- 替代方案：用原型 HTML 直接替换现有页面，或同步重写数据模型与问卷逻辑。
- 影响：首页、问卷页和结果页获得黑白灰报告式 UI、共享导航和移动端全屏菜单；原型中的粒子文字、复杂滚动动画和更高阶图形化先留到后续视觉窗口复核。

- 日期：2026-06-25
- 决策：P0 纠偏后，UI 不再走“近似整合/二次设计”路线，改为直接将 `ui-prototypes/` 中已完成的静态 HTML/CSS/JS 产品化为 Next.js/React 页面。
- 原因：用户明确要求使用已经做好的界面，而不是重新设计、抽象设计语言或还原一张相似页面；产品需要以静态原型作为视觉和交互验收基准，只替换必要业务接口。
- 替代方案：继续在现有组件上做黑白灰视觉优化；或完全复制 HTML 并放弃 React 状态和现有业务链路。
- 影响：新增 `app/prototype.css` 作为原型样式基准，首页、问卷页、结果页改为原型 DOM 结构承载真实数据；评分、题库、结果生成和 localStorage 规则保持不变。后续 UI 改动必须先对照 `ui-prototypes/`，避免再次偏离。

- 日期：2026-06-25
- 决策：首页 `/` 直接读取并渲染 `ui-prototypes/human-3-ui-v2.html` 的原始 `<style>` 和 `<body>`，不再通过 React 组件复刻首页。
- 原因：用户实际查看后指出首页差异仍大，并明确要求“直接套用原文件”；首页视觉一致性优先级高于组件抽象。

- 日期：2026-06-25

- 原因：用户希望后续能按“页面接接口”继续开发；把评分和分享能力抽成接口可以稳定页面边界，但当前 MVP 没有数据库和登录，不能让临时 API 失败阻断用户完成评估。
- 替代方案：继续全部在前端本地生成结果；或强制所有流程只走 API、失败即报错。
- 影响：新增 `POST /api/assessment/score`、`POST /api/share/encode`、`POST /api/share/decode`；`/assessment`、`/result/share` 和复制分享链接已优先调用 API，失败时本地回退。评分、题库、结果模板和 localStorage 规则不变。
